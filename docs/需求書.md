## 0. 專案背景與現有資產

- 業主為嘉義的預約制經絡調理工作室，共兩位調理師，行事曆與系統中一律以代號稱呼：**神藏（鈞）**、**神庭（萱）**。
- 客群包含長者與術後／中風後調理客戶，介面與訊息設計需以「操作極簡、字體清楚、步驟少」為原則。
- 現有資產（開發時優先沿用，避免重複建置）：
  - LINE 官方帳號（已有客戶好友基礎，現行以人工一對一對話預約）
  - 共用 Google 行事曆「**鈞&萱**」：現行預約以薰衣草色標記，事件中以神藏／神庭區分調理師（實際事件命名格式請於開發初期向業主確認並據此設計解析規則）
  - GitHub Pages 靜態網站部署經驗（既有客戶紀錄 web app：`https://a0985180305-dotcom.github.io/relax/index.html`）
  - Cloudflare Worker（既有 AI proxy 使用經驗）
  - Firebase / Firestore（既有客戶紀錄資料庫，已啟用驗證式安全規則）
  - 既有紙本／表單「經絡調理表」，內含客戶基本資料欄位（首次預約的必填資料以此為準，向業主索取欄位清單）
- 系統目標：客戶在 LINE 內自助查詢空檔並送出預約；後台以「**系統建議、人工核准**」為原則管理時段（第一版不做全自動上架）；預約即時同步 Google 行事曆；以 LINE 推播完成通知與行前提醒。

## 1. 名詞定義（全文件依此為準）

| 名詞 | 定義 |
|---|---|
| 節 | 一個調理單位 = **50 分鐘** |
| 時段單位 | 所有開始時間以 **10 分鐘**為單位（如 09:10 可，09:15 不可） |
| 間隔（buffer） | 任一預約與前後既有行程之間固定保留 **20 分鐘** |
| 營業時間 | 每日 **07:00–12:00、14:00–21:00**（Asia/Taipei） |
| 調理師 | 神藏、神庭 |
| 空檔時段（slot） | 屬於某位調理師、對外可被預約的一個開始時間，附可容納節數 |
| 當日已排節數 | 該調理師當日狀態為「待確認」＋「預約成功」的節數總和 |
| 取消管控觸發條件 | 符合**任一**條件即觸發：(a) 連續兩筆預約皆取消，且兩筆之間無任何「已完成」出席紀錄；(b) 30 天滾動窗口內累積取消 3 次 |
| 不指定調理師 | 客戶預約時的第三種選項，折價 100 元，由系統依規則指派調理師（見 3.1） |
| 預約截止時間 | 每日 **21:00** 關閉所有**隔日**的可預約時段；此後隔日時段僅能透過專人對話預約（實作上以查詢時過濾即可，效果等同下架） |
| 雙人同行預約 | 主預約人為自己與一位親友同時預約，兩位調理師同時段分別調理（見 3.5），資料上為兩筆預約以群組 ID 綁定 |
| 方案客 | 客戶檔案中有**有效方案**（剩餘節數 > 0 且效期未過）者；方案用完或過期即自動回到散客身份 |

## 2. 預約狀態機

```
開放中 ──客戶送出──▶ 鎖定待確認 ──管理員核准──▶ 預約成功 ──▶ 已完成 / 未出席
                        │                              │
                        └─管理員拒絕──▶ 釋放重開或關閉    └─客戶取消(期限內)──▶ 已取消(時段自動重開)
```

- 客戶按下預約的**瞬間**，該時段對外即時隱藏（防止重複預約），並同步在 Google 行事曆建立事件，標題前綴【待確認】。
- **鎖定不設自動過期**，維持鎖定直到管理員人工核准或拒絕。
- 核准後：行事曆事件轉為正式格式（沿用現行薰衣草色與命名慣例），推播客戶「預約成功通知」。
- 拒絕後：詢問管理員該時段要「重新開放」或「關閉」，行事曆待確認事件刪除。
- 取消期限：**前一日 21:00 前**。期限內客戶可自助取消，時段自動重新開放並即時通知管理員；期限後不開放自助取消，引導聯繫專人。

## 3. 客戶端（LINE LIFF）

### 3.1 查詢空檔與預約
1. 進入預約頁 → 選擇預約類型：**單人預約／雙人同行（親友）**（雙人流程見 3.5）。
2. 選擇調理師，三個選項：**神藏／神庭／不指定調理師（可折價 100 元）**。
   - 首次預約由客戶主動選擇；第二次起自動帶入上次選擇，仍可修改。
   - 選「不指定」時：顯示的空檔為兩位調理師開放時段的**聯集**（同一時間兩位皆開放時去重顯示一筆）。系統於客戶送出預約當下依下列順序指派並鎖定該調理師的時段：
     1. 當天空檔較多的調理師；
     2. 若當天時數相同 → 當週已預約時數較少者；
     3. 若仍相同 → 未曾服務過該客戶者；
     4. 若仍無法區分（皆服務過或皆未服務過）→ 系統隨機指派。
   - **保密規則**：管理員通知中必須註明「客戶選擇不指定」及系統指派的調理師；但**客戶端的所有訊息（成功通知、行前提醒）一律不顯示調理師名稱**，調理前不讓客戶知道由誰服務。
3. 選擇節數：**1–4 節，預設選項為 2 節**（介面上將 2 節標示為推薦）→ 顯示未來 **4 周**內可容納該節數（含前後間隔）的開放時段。
4. **預約截止時間**：每日 21:00 起，隔日所有時段不再顯示、不可預約（與取消期限為同一時間點，方便客戶記憶）；21:00 後如需預約隔日，LIFF 頁顯示引導文案改以專人對話處理。
5. 送出預約 → 時段即時鎖定 → 以 **Reply API** 回覆「已收到您的預約申請，確認後將另行通知」（不耗推播額度）。

### 3.2 首次預約資料填寫
- 以 LINE userId 綁定客戶檔案。首次預約需填寫基本資料（欄位沿用既有「經絡調理表」的基本資料部分）。
- **個人狀況補充欄位**（每次預約皆顯示，偏好類選項存入客戶檔案、下次自動帶入可修改）：
  - ☐ 我會穿寬鬆或有彈性的服裝
  - ☐ 我需要借用服裝現場更換（酌收清潔費 50 元）
  - ☐ 我怕冷需要電熱毯
  - ☐ 我會自備鋪蓋於床墊的大浴巾及小毛巾（可享折扣 50 元）
  - 其他補充：＿＿＿＿（自由填寫文字框）
- 費用加減項（±50 元）僅記錄並顯示於雙方通知中，**不涉及線上金流**，現場結算。

### 3.3 通知訊息（模板參考業主提供之範例影片模式）
1. **預約成功通知**（Push）：日期時間、調理師、節數與時長、工作室地址、行前注意事項（穿著寬鬆有彈性的服裝、自備毛巾者請攜帶大浴巾＋小毛巾等，依客戶勾選的補充欄位動態調整內容）、**「加入行事曆」.ics 連結**。若為「不指定調理師」的預約，通知中**不顯示調理師名稱**。
2. **行前提醒**（Push，前一日發送，建議 18:00–19:00 間）：提醒明日時段與注意事項，附快速回覆按鈕：
   - 「✅ 準時出席」→ Reply 感謝訊息，記錄已確認。
   - 「🔁 需要改期」→ Reply 流程列出未來 4 周可預約時段供直接改約；改約成立後原時段釋放並通知管理員。
   - **未回應 → 視同出席**，不取消，僅記錄。

### 3.4 取消與資格管控
- 期限內自助取消：時段自動重開。客戶**第一次取消時**，隨取消確認一併推播提醒：「提醒您：若連續取消預約，或 30 天內取消達 3 次，線上預約功能將暫停 7 天，期間需透過專人對話預約。」
- **觸發取消管控**（符合第 1 節定義之任一條件）：
  - 停用該客戶的線上預約功能 **7 天**，期滿自動恢復；
  - 停用期間 LIFF 頁顯示引導文案，改以官方帳號一對一對話由管理員手動代為預約；
  - 觸發時推播通知客戶（說明原因與恢復日期）並通知管理員；
  - 管理員可於後台手動提前恢復其資格。

### 3.5 雙人同行預約
- 適用情境：主預約人為自己與**一位**親友預約同一時段，由兩位調理師同時分別調理。
- 流程：
  1. 於 3.1 第 1 步選「雙人同行」→ 選擇節數（**兩人強制同節數**，v1 不支援不同節數）。
  2. 時段清單改為顯示**兩位調理師同時開放且皆可容納該節數的時段（交集）**，標示為「雙人時段」。
  3. 調理師配對：主預約人可選「**指定誰服務我**（另一位自動服務同行親友）」或「**由工作室安排配對**」，配對選擇不影響折價。
  4. **雙人同行優惠：每人折 100 元（每筆預約計一次，非每節），一組共折 200 元**——此為促購優惠，LIFF 於選擇「單人／雙人同行」的步驟即應顯示「雙人同行每人折 100 元」文案，吸引原本單人預約的客戶攜伴。單人預約的「不指定調理師 −100」規則維持不變，兩者不疊加（雙人同行固定各 −100，與配對方式無關）。
     - **方案客不適用雙人同行折價**（方案價已優惠）：折價資格**以主預約人的方案狀態判定整組**。主預約人為有效方案客 → 整組（含同行親友）皆不顯示折價文案與金額（方案客通常直接以方案堂數為親友買單）；主預約人為散客 → 兩人各 −100。
     - 折價顯示（客戶通知、管理員摘要）一律加註：「優惠以現場結算為準」，避免方案欄位手動維護的時間差造成爭議。
  5. 主預約人**代填同行親友資料**：姓名＋與 3.2 相同的個人狀況補充複選欄位（親友通常未加入官方帳號，所有通知一律發給主預約人）。
- 資料模型：兩筆預約各自佔用一位調理師的時段，以**群組 ID（group_id）綁定**；此欄位須於 Phase 1 的資料模型即預留。
- 通知：成功通知與行前提醒發給主預約人，內容涵蓋兩人；管理員通知標示為雙人同行並列出配對結果。
- 取消：可**整組取消**或**僅取消一人**（僅取消一人時另一人保留，被釋放的調理師時段自動重開）；取消紀錄與取消管控一律計在主預約人。
- 計費：兩人各自依節數照原價計算，現場結算。

## 4. 後台（管理端）

### 4.1 建議時段引擎（核心演算法，**只產生建議清單、不自動上架**）

**硬規則（絕不違反）：**
1. 僅在營業時間內，且整個時段（含節數）不得跨越午休（12:00–14:00）或超出 21:00。
2. 開始時間以 10 分鐘為單位。
3. 與該調理師前後既有行程（含待確認、成功、行事曆手動行程）間隔 20 分鐘。
4. 含間隔已連續排滿 **6 節**後，至少間隔 **60 分鐘**才可再開下一個時段。
5. **每位調理師每日對外同時開放的空檔時段數量上限**（依當日已排節數動態計算）：
   - 已排 < 4 節 → 最多 2 個開放時段
   - 已排 4–7 節 → 最多 1 個
   - 已排 ≥ 8 節 → 0 個（當日不再開放）
   - 開放時段被預約成功後，引擎可建議遞補新時段，仍受本上限約束。

**軟偏好（產生候選後依此權重排序，權重由高至低）：**
1. 兩位調理師工作時間**重疊**：優先建議「另一位調理師同時段有服務」的時間；次之為相近時間。
2. 同一位調理師的預約**連續**：緊接在既有預約之後 ＞ 緊接在既有預約之前。
3. 時間帶偏好：避開上午 11:00 後開始、晚上 20:00 後、早上 08:00 前；**07:00–08:00 僅在該調理師當日已無其他可建議時段時才列入建議**。

軟偏好互相衝突時依上列順序取捨；硬規則永遠優先於軟偏好。

### 4.2 核准工作流
1. **每周日**產生未來 4 周的建議時段清單，推播管理員並於後台頁面呈現，逐筆或批次核准後才上架對外開放。
2. 周間如有新空檔出現（客戶取消、行事曆行程變動、預約成功後的遞補建議），引擎**即時以建議形式通知管理員**，核准後上架；期間系統不自行新增任何對外時段。
3. 客戶送出預約時，**即時推播兩位管理員**：預約摘要（客戶、調理師、時間、節數、補充欄位內容）＋核准／拒絕操作連結＋「是否開放遞補時段」的建議清單。若客戶選擇「不指定調理師」，摘要中須明確標示**「客戶選擇不指定」及系統依規則指派的調理師與指派理由**（例：當天空檔較多／當週時數較少／未曾服務）。

### 4.3 Google 行事曆同步
- 「待確認」與「預約成功」事件即時寫入共用行事曆「鈞&萱」，以標題前綴或色彩區分狀態（正式預約沿用現行薰衣草色慣例）。
- **偵測行事曆手動新增／變動的行程**：優先使用 Google Calendar API push notifications（watch channel）；若不可行則輪詢，間隔不超過 5 分鐘。偵測到與開放時段衝突（含 20 分鐘間隔）時，即時關閉該時段並通知管理員。

### 4.4 客戶資料
- 以 LINE userId 為鍵綁定客戶檔案；請評估與既有 Firebase 客戶紀錄 app 的整合或並存方案，向業主提案。
- 客戶檔案包含：基本資料、偏好（怕冷／自備毛巾／服裝等）、預約與出席歷史、取消紀錄與線上預約資格狀態、**方案欄位**（方案類型、剩餘節數、到期日）。
- **方案扣節與剩餘次數管理已由既有客戶紀錄系統負責**：該系統會在調理完成後扣除次數，並以 LINE 推播通知客戶剩餘次數與效期（格式範例：「本次調理已完成，已扣除 1 次。方案『買10送2』剩餘 N 次，效期至 YYYY/MM/DD。」）。本預約系統 **v1 不重複實作扣節邏輯**，方案欄位僅供唯讀顯示或由管理員手動同步；兩系統的資料整合列入未來 Phase 4。

## 5. 通知設計與 LINE 訊息額度

- 原則：**客戶觸發的回應一律用 Reply API（免費）**；只有主動通知才用 Push（計費）。
- 每筆預約的 Push 預算：客戶 2 則（成功通知＋行前提醒）＋管理員 2 則（兩位各 1 則預約待核准通知）；周日建議清單與周間遞補建議另計。**注意：既有客戶紀錄系統每次調理完成也會發 1 則扣節通知（同一個官方帳號額度）**，估算總量時須合併計入。
- 業主已決定採用 **LINE OA 中用量方案（3,000 則／月）**，開發時仍請在後台提供當月 Push 用量計數，便於監控。
- 管理員通知以 LINE 推播為主；行事曆即時寫入天然構成備援通知管道。

## 7. 技術建議（可由開發者評估調整，變更請說明理由）

- **前端**：LINE LIFF 頁面，託管於 GitHub Pages 或 Cloudflare Pages；管理端後台為簡單的驗證保護網頁。
- **後端**：Cloudflare Worker（LINE Messaging API webhook、Google Calendar API、Cron Triggers 處理周日批次與行前提醒排程）。
- **資料庫**：優先評估沿用既有 Firestore 專案；Cloudflare D1 或 Supabase 為備選。
- **Google Calendar API**：OAuth refresh token 存於 Worker secrets；絕不將任何金鑰寫入前端或版本庫。
- **時區**：所有時間運算一律 Asia/Taipei，注意 LINE 計費以日本時區結算（台灣 23:00 後發送計入次日）。
- **安全**：LIFF 端以 LINE idToken 驗證身分；管理端 API 需驗證；Firestore 規則比照既有專案的驗證式規則。

## 8. 驗收情境（每階段完成時逐項演示）

1. 客戶選神藏、2 節，看到的時段皆能容納 100 分鐘＋前後 20 分鐘間隔，且不跨午休。
2. 兩位客戶同時點同一時段，僅一人成功鎖定，另一人看到時段已消失。
3. 管理員核准後，客戶收到成功通知含 .ics 連結，行事曆事件由【待確認】轉正式。
4. 行事曆手動新增一筆行程後 5 分鐘內，衝突的開放時段自動下架。
5. 神藏當日已排 8 節時，引擎不再為其建議任何當日新時段。
6. 客戶於行前提醒按「需要改期」，完成改約，原時段釋放且管理員收到通知。
7. 客戶連續第 2 次取消（或 30 天內第 3 次取消）後，線上預約停用 7 天：LIFF 頁顯示暫停文案與恢復日期，第 8 天自動恢復。
8. 客戶選「不指定調理師」：管理員通知標示指派結果與理由；客戶收到的成功通知與行前提醒皆不出現調理師名稱；指派順序符合「當天空檔多 → 當週時數少 → 未服務過 → 隨機」。
9. 當日 21:01 查詢，隔日所有時段不顯示且不可預約，LIFF 顯示專人預約引導；20:59 查詢則隔日時段正常可約。
10. 雙人同行預約：時段清單僅出現兩位調理師皆開放的交集時段；送出後兩筆預約以群組 ID 綁定、各佔一位調理師時段；僅取消其中一人時，另一人保留且被釋放的時段自動重開。

## 9. 分階段交付

> 每階段開始前，先提交該階段的架構與資料模型設計供業主確認；完成後依第 8 節演示驗收，再進入下一階段。

- **Phase 1｜預約核心**：LIFF 查空檔＋預約鎖定＋預約截止時間（3.1 第 4 點）＋首次資料填寫＋管理員核准流程＋行事曆雙向同步＋預約成功通知（含 .ics）。時段先以管理員手動建立為主。**資料模型須於本階段即預留雙人同行的群組 ID 欄位。**
- **Phase 2｜排程建議引擎**：4.1 全部規則＋周日批次建議＋周間遞補建議＋手動行程偵測關閉。
- **Phase 3｜生命週期自動化**：行前提醒與出席確認＋改期流程＋自助取消與時段自動重開＋取消資格管控＋**雙人同行預約（3.5）**。
- **Phase 4（未來，不在本次範圍）**：與既有客戶紀錄系統（含扣節與效期管理）的資料整合、成效統計報表。
