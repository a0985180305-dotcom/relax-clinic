# 產品化藍圖：把單店系統變成可以賣的產品

> 這份文件是「方向決策」，不是施工圖。目的：現在把路選對，之後任何模型（Sonnet 即可）都能照著施工。
> 撰寫日期：2026-07-04。基於當時程式碼盤點（commit 4666fe6 之後）。

---

## 1. 產品定位（一句話）

**給非工程背景的調理／美容／身心工作室：「客戶加 LINE 就能預約、買方案自動管點數」的一站式系統，老闆不用懂任何技術。**

目標客群跟你自己一模一樣——這是最大優勢：你踩過的每個坑、每個「這裡好難懂」，都是產品的改進清單。

---

## 2. 商業模式：三選一

| 模式 | 做法 | 優點 | 缺點 |
|---|---|---|---|
| A. 賣斷 | 收一次錢，幫對方架好，之後不管 | 簡單 | 對方出問題還是會找你，等於免費售後 |
| B. 代管訂閱 ⭐ | 導入費＋月費，系統掛在你手上，你負責維運更新 | 持續收入；客戶是調理師、本來就不想碰技術 | 你要扛維運（但見第 7 節，成本近零） |
| C. 純軟體 SaaS | 做成自助註冊平台 | 最能規模化 | 工程量是現在的 10 倍，一個人做不動 |

**建議：B 代管訂閱制。** 理由：你的客群「付錢買省事」的意願遠高於「付錢買軟體」。定價方向見第 7 節。等付費店家超過 10 家、月費收入穩定，再評估往 C 演進。

---

## 3. 最關鍵的架構決策：每店一套 vs 共平台

這是「一開始選錯、之後全部重做」的那個決定。

### 先認清一個改不掉的硬限制：LINE

客戶加的是「那家店」的 LINE 官方帳號。所以**每一家店都必須有自己的**：
- LINE 官方帳號（Messaging channel，推播用）
- LINE Login channel ＋ LIFF（預約頁身分驗證用）

這部分無論選哪條路都省不掉，每開一家店都要走一次 LINE 申請流程。這反而是好事——它天然決定了「每店一套設定」的格局。

### 兩條路比較

| | 每店一套（模板複製） | 共平台（多租戶） |
|---|---|---|
| 做法 | 每家店：一個 Firebase 專案＋一個 Worker＋一份前端 | 一套系統，資料庫用「店家 ID」分隔所有資料 |
| 資料隔離 | 天然隔離，A 店絕不可能看到 B 店病歷 | 靠程式邏輯隔離，寫錯一行就是跨店洩漏病歷 |
| 免費額度 | 每店各自享有 Firebase／Cloudflare 免費額度 | 全部店家共吃一份額度，量大要付費 |
| 出事影響範圍 | 一家店掛，只影響一家 | 平台掛，所有店一起掛 |
| 改一個 bug | 要逐店更新（N 家店改 N 次） | 改一次全部生效 |
| 工程難度 | 幾乎零（現有程式碼直接用） | 要重寫資料結構＋安全規則＋權限系統 |

### ✅ 決定：Phase 1 走「模板複製」，並用以下方法解決它唯一的痛點

模板複製唯一的缺點是「改 bug 要逐店更新」。解法：

1. **本 repo 就是母版（template）**。所有功能開發、修 bug 都只在母版做。
2. 每家新店 = 母版的一個複本（GitHub 上 fork 或另開 repo），**店家差異只存在於設定區塊**（見第 4 節）。
3. 更新流程：母版修好 → 逐店把新版檔案覆蓋過去、保留該店設定區塊 → 完成。因為差異只有設定區塊，這個動作可以請 AI 半自動化。
4. 病歷資料在各店自己的 Firebase 專案裡，更新程式碼完全不碰資料。

**Phase 2 觸發條件**（同時滿足才考慮重寫成共平台）：付費店家 > 10 家、月費收入足以支付工程外包或大量 AI 開發時間、逐店更新每月耗時超過一個工作天。

## 4. 現在就該做的事：去硬編碼（本次盤點結果）

目標：**開新店時，只需要改每個檔案最上面的一個「店家設定區塊」，其他一行都不用動。**

目前寫死的東西和位置：

| 檔案 | 寫死的內容 | 位置 |
|---|---|---|
| `index.html` | firebaseConfig（專案 ID 等） | 約 567–570 行 |
| `index.html` | WORKER_URL、LIFF_ID | 約 579、581 行 |
| `index.html` | PLAN_CONFIG 方案規則 | 約 583–585 行 |
| `index.html` | 調理師代號 `glan`／`xuan` | 散落約 50 處 ⚠️ |
| `booking.html` | LIFF_ID、WORKER_URL | 約 139–140 行 |
| `booking.html` | 調理師代號 `glan`／`xuan` | 散落約 11 處 ⚠️ |
| `booking-worker.js` | PLAN_CONFIG（與 index.html 重複一份！） | 約 28–30 行 |
| `wrangler.toml` | FIREBASE_PROJECT_ID、Worker 名稱 | 15 行附近 |

改造工單（可交給 Sonnet 逐項做，每項做完都要走三層驗收）：

- [ ] **工單一：抽出 THERAPISTS 設定。** 在 `index.html` 與 `booking.html` 檔案頂部建立 `THERAPISTS = { glan: { name: '鈞' }, xuan: { name: '萱' } }` 設定物件，把散落 61 處的寫死代號改為讀取設定。這是工程量最大的一項，也是新店客製（調理師人數／名字不同）的關鍵。**注意：這會動到已上線檔案，先建 commit 還原點，改完要完整回歸測試。**
- [ ] **工單二：PLAN_CONFIG 單一來源。** 目前 `index.html` 和 `booking-worker.js` 各有一份，哪天只改了一邊就會出現「後台顯示 12 次、Worker 算 10 次」的災難。建議搬進 Firestore `config/app`（pushSecret 已經在那裡，模式現成），兩邊開機時讀取。
- [ ] **工單三：店家設定區塊標準化。** 每個檔案頂部整理出統一格式的 `/* ===== 店家設定（開新店只改這裡） ===== */` 區塊，包含：firebaseConfig、WORKER_URL、LIFF_ID、店名、營業時段預設值。
- [ ] **工單四：SETUP.md 升級為「開店手冊」。** 按第 5 節的 SOP 骨架重寫，標準是「一個完全沒碰過的調理師，照著做能自己開起來」——做不到沒關係，因為代管模式下是你來操作，但寫到這個標準能逼出所有隱藏步驟。

## 5. 開一家新店的 SOP 骨架（未來開店手冊的目錄）

1. 向店家收集：店名、調理師名單、方案規則（買幾送幾、效期）、營業時段
2. LINE：申請官方帳號 → 建 Messaging channel → 建 Login channel ＋ LIFF（貼上該店預約頁網址）
3. Firebase：開新專案 → 開 Firestore → 貼安全規則 → 建 `config/app`（pushSecret 等）
4. Cloudflare：複製 Worker → 改 wrangler.toml 名稱與專案 ID → 設 secrets → 部署
5. 前端：複製母版 repo → 填店家設定區塊 → 推上 GitHub Pages
6. 串接驗證：跑一遍「加好友 → 綁定 → 預約 → 確認 → 完成扣點 → 收到推播」全流程
7. 交付店家：後台網址＋登入方式＋一頁式使用說明

每一步未來都值得配截圖。目前先有骨架，第一次幫別人開店時邊做邊補血肉。

## 6. 每家店的暫定命名規約

開到第三家店就會亂，現在先定：

- Firebase 專案 ID：`mclinic-{店家代號}`（例：`mclinic-shenting`）
- Worker 名稱：`booking-api-{店家代號}`
- GitHub repo：`clinic-{店家代號}`（⚠️ 記得開 **private**——母版目前是 public，新店 repo 不該讓人看到店家設定）

> 順帶提醒：母版 repo 目前是 public。程式碼公開沒問題（裡面沒有機密），但本藍圖若含定價策略等商業內容，要考慮是否改 private 或把商業章節另存本機。

## 7. 成本結構與定價方向

**每家店的邊際成本（以目前用量等級）：**

| 項目 | 費用 |
|---|---|
| Firebase Spark 免費方案 | $0（單店讀寫量遠低於免費額度） |
| Cloudflare Workers 免費方案 | $0（每日 10 萬次請求，單店用不到 1%） |
| GitHub Pages | $0 |
| LINE 官方帳號輕用量 | $0（每月 200 則免費推播；超過由店家自己升級，約 NT$800/月，這筆讓店家自付） |

也就是說：**多接一家店，你的月成本幾乎是零**，收的月費扣掉你的時間就是淨利。

**定價方向（待市場驗證，先給錨點）：**
- 導入費：一次性，涵蓋你走完第 5 節 SOP 的工時（LINE 申請最耗時）
- 月費：涵蓋維運、更新、客服；對標店家「省下的紙本／人工排約時間」而非軟體行情
- LINE 推播超量費：明訂由店家自付，避免日後糾紛

具體數字等第一家試點店談完再定。**建議第一家找熟識同業，用極低價換完整回饋＋見證。**

---## 8. 風險與待決事項

| 風險 | 對策 |
|---|---|
| **個資責任**：別家店的客戶病歷在你代管的系統裡 | 服務條款要白紙黑字：資料所有權歸店家、你是受託處理；每店資料在獨立 Firebase 專案（模板複製架構天然做到）；正式簽約前諮詢一次法律意見 |
| **你是單點故障**：系統只有你會維護 | 開店手冊寫到位＝任何人（或任何 AI）能接手；母版 repo 就是完整交接文件 |
| **客服量失控**：店家半夜密你「客人約不進來」 | 月費要把客服時間算進去；建 FAQ；明訂支援時段 |
| **LINE 政策變動**：LIFF／免費額度規則改版 | 無法避免，但每店獨立 channel 使影響分散；留意 LINE 開發者公告 |

- [ ] 母版 repo 是否轉 private？（若要放商業內容則建議轉）

- [ ] 導入費／月費數字
- [ ] 品牌名稱（產品要有自己的名字，不叫「神庭&神藏系統」）

## 9. 建議施工順序（全部可用 Sonnet 執行）

```
現在（系統收尾期）
 ├─ 1. 收尾：班表、Login channel 發布、圖文選單   ← 自己店先全自動
 ├─ 2. 工單二：PLAN_CONFIG 單一來源                ← 順手消掉同步風險
 └─ 3. 到期提醒推播、週報表                        ← 自己店先賺到自動化的甜頭

醞釀期（自己用 1–2 個月、系統穩定後）
 ├─ 4. 工單一：THERAPISTS 設定抽離（大工程，排穩定期做）
 ├─ 5. 工單三＋四：設定區塊標準化、開店手冊
 └─ 6. 物色第一家試點店

試點期
 ├─ 7. 照開店手冊幫試點店開起來（邊做邊修手冊）
 └─ 8. 收集回饋、定價定案 → 開始對外
```

原則：**每一步都先讓自己的店受益**，產品化是副產品，不是額外負擔。
