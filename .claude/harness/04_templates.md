# 04｜標準化派工 Prompt 模板

> 用法：主對話要派 subagent 時，複製對應模板、把 `〔〕` 填滿後作為 Agent 的 prompt。
> 通則：subagent 是**冷啟動**，沒讀過主對話任何內容——所有背景都要寫進 prompt。
> `〔〕` 一個都不准留空；填不出來代表你自己還沒想清楚，先想清楚再派工。

---

## 模板 1｜深度搜尋與研究（Search & Research）

**Agent 類型**：`Explore`（唯讀，**僅限本機檔案搜尋**）；凡需要 WebFetch／WebSearch／curl 的線上查證或跨網研究，一律改用 `general-purpose`。**model**：sonnet（預設）／haiku（純定位檔案）

```
【背景】本專案是經絡調理診所的 LINE 預約系統。先讀專案根目錄 CLAUDE.md 的
「系統全貌」與「事實來源對照表」（不用讀其他段落）。

【目標】找出：〔要找什麼，一句話，例如「index.html 中所有呼叫 Worker API 的位置與 action 名稱」〕
【為什麼】〔主對話拿這結論要做什麼，一句話〕

【搜尋範圍】〔檔案/目錄，例如 index.html 與 booking.html〕
【搜尋深度】〔medium／very thorough〕

【驗收條件】
- 每個結果都有 檔名:行號
- 沒找到也是合法答案，寫「未找到」＋你搜過哪些關鍵字，禁止硬湊

【回報格式（嚴格遵守）】
- 條列，每條＝「檔名:行號 — 一句話說明」
- 最後給 ≤5 行的整體結論
- 禁止貼出任何超過 3 行的代碼片段
```

## 模板 2｜新功能實作（Implementation）

**Agent 類型**：`general-purpose`。**model**：sonnet；被退回 2 次後改 opus（附錯誤軌跡，見 02）

```
【背景】本專案是經絡調理診所的 LINE 預約系統。動手前必讀：
1. 專案根目錄 CLAUDE.md（全文，特別是紅線與環境陷阱）
2. .claude/harness/01_diagnostics.md（三大禁令）
3. 〔本任務相關的既有代碼位置，例如「index.html 的班表區段，Grep 'loadSchedule' 定位」〕

【目標】〔功能描述＋使用者故事，例如「後台客戶列表加『匯出 JSON』按鈕，點擊下載全部客戶基本欄位（不含病歷內文）」〕
【明確不做】〔範圍外的事，例如「不改動既有列表渲染邏輯、不動 Worker」〕

【硬性約束】
- 改前先 git commit 建還原點
- 只做局部 Edit，禁止重寫整檔/整函式群；>300 行的檔案先 Grep 錨點再讀區段
- 禁止 git push、禁止部署、禁止呼叫清單外 MCP 工具

【驗收條件 AC】（逐條可機器判定）
1. 〔例：node --check 通過（hook 會自動驗）〕
2. 〔例：Grep 'exportClients' 在 index.html 恰好出現於函式定義與按鈕綁定兩處〕
3. 〔例：新增代碼不含任何寫死的密鑰或客戶個資〕

【回報格式】
- 改了哪些檔：檔名:行號範圍
- AC 逐條核對結果（過/不過＋證據）
- 「使用者 3 步驗證清單」草稿（上線後用手機/瀏覽器可操作）
- 未做/風險清單（可為空但必寫）
- 禁止貼大段代碼，主對話會自己看 diff
```

## 模板 3｜架構重構（Refactoring）——先計畫後動工

**Agent 類型**：`Plan`（先出計畫）→ 使用者同意 → `general-purpose` 實作。**model**：opus 出計畫，sonnet 執行

```
【背景】同模板 2 的必讀清單。另外必讀：git log --oneline -10（哪些是已上線驗證的 commit）。

【目標】〔重構動機與範圍，例如「把 index.html 內散落的 Worker URL/常數集中到單一 config 區塊」〕

【禁區（碰到即失敗）】
- 不得改變任何對外行為（API 形狀、UI 動線、Firestore 結構）
- 不得動 〔明列已上線且與本次無關的區段〕
- booking-worker.js 的驗簽/扣點邏輯屬核心，重構它一律需使用者逐項同意

【產出（先計畫，不寫碼）】
1. 要動的檔案與區段清單（檔名＋Grep 錨點）
2. 每一步的「改動→驗證」配對（改一步驗一步，不准最後一次驗）
3. 回滾方案（哪個 commit 是還原點）

【計畫經使用者同意後才實作。實作時 AC】
1. 重構前後行為一致的證明方法：〔例：對同一組測資，重構前後輸出 diff 為空〕
2. hooks 全綠、git commit 分步提交（每步一 commit）
```

## 模板 4｜代碼與安全審查（Code Review）

**Agent 類型**：`general-purpose`（唯讀立場，禁止動手改）。**model**：opus（審查是弱模型的相對強項但涉安全用高階）

```
【背景】本專案 repo 是公開的；系統服務真實客戶，含病歷（Firestore）與 LINE 身分驗證。
先讀 CLAUDE.md 的紅線與關鍵常數，再審以下範圍。

【審查範圍】〔例：git diff HEAD~1，或指定檔案〕

【審查重點（按優先序）】
1. 密鑰/個資洩漏：任何 token、私鑰、客戶資料出現在代碼或註解中
2. 驗證繞過:Worker 的 LINE id token 驗證、pushSecret 比對是否可被繞過
3. 正確性:點數扣減、效期計算、時區處理的邊界錯誤
4. 其次才是:重複代碼、可簡化處

【硬性規則】
- 只報告，不修改任何檔案
- 每個發現必附:檔名:行號＋嚴重度(高=洩漏/資損可能、中=行為錯誤、低=品質)＋一句話修法建議
- 不確定是否為問題的,標「待查證」並寫明查證方法,禁止腦補定罪

【回報格式】
- 按嚴重度分組的表格(檔名:行號｜嚴重度｜問題｜建議)
- 結尾一句總評:可上線/需修高風險項/需使用者決策
```
