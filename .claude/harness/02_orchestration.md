# 02｜多模型調度與動態升降級守則

> 前提現實（先弄清楚，別照幻想操作）：主對話用哪個模型是**使用者在 App 裡選的**，模型無法自行切換主對話模型。但任何主模型都能用 Agent 工具派 subagent，並以 `model` 參數指定 `haiku`／`sonnet`／`opus`。派工模板見 `04_templates.md`。

## 名詞定義（全 harness 統一，重試次數一律以本段為準）

- **工具層失敗**＝單一工具或指令調用出錯一次。同一工具、**同一種錯誤**出現第 2 次＝「兩振出局」→ 停手換路徑（`03_rubrics.md` R1）。Haiku 特例：第 1 次就升級改派，見下表。
- **任務層一輪**＝對同一子任務「一次完整嘗試＋一次修正」。同一子任務最多**兩輪**，第三輪不存在 → 熔斷問使用者（`03_rubrics.md` R3）。
- 換句話說：小失誤在工具層數到 2，整件事在任務層數到 2 輪，兩個計數器互不混用。

## 原則 0：先問「要不要派」（防儀式性派工浪費使用者的額度）

本專案規模小。單檔小修、一兩處 Edit、回答問題 → **直接做，不派工**。
只有三種情境需要派 subagent：
- **A. 大量讀檔／全 repo 掃描**（保護主對話 context 不被塞爆）→ 派 `Explore`（唯讀）
- **B. 隔離驗證**（見下方「隔離驗證」）→ 派 fresh-context 驗收員
- **C. 連續卡關升級**（見下方「升降級路徑」）→ 派高階模型救援

## 指揮官不下場

主對話＝指揮官：定目標、拆任務、收結論、對使用者負責。
大批量讀檔、搜尋、掃描一律指派 subagent；subagent 回報只准精簡結論（路徑＋行號＋一句話），**主對話不接收大段代碼**。指揮官親自下場讀 87KB 的 index.html，就是 `01_diagnostics.md` 痛點 2 的開端。

## 派工三件套（缺一不發）

1. **目標與背景**：要達成什麼＋為什麼＋必讀檔案清單（至少含 `CLAUDE.md`；subagent 是冷啟動，沒讀過本對話任何內容）。
2. **驗收條件 AC**：可機器判定的清單，例如「`node --check` 通過」「Grep 得到 X 於某檔」「WebFetch 線上頁含某字串」。禁止「做好做滿」「確保品質」這類空話。
3. **回報格式**：成果路徑＋關鍵行號＋不超過 10 行摘要。禁止在回報中噴出大段代碼。

## 升降級路徑

| 執行者 | 觸發條件 | 動作 |
|---|---|---|
| Haiku／小模型 | 工具呼叫錯誤或語法錯誤 **1 次** | 同一任務直接改派 Sonnet，不给第二次機會 |
| Sonnet／中模型 | 同一子任務卡關或出錯 **2 次** | 整理「錯誤軌跡」（格式見下），改派 `model: opus` 的 subagent |
| Opus 也解不動，或已重試兩輪 | — | **熔斷**：停止自主作業，白話回報使用者（判準：`03_rubrics.md` R3） |
| Opus 解出固定模式 | 同類問題日後會再現 | 把解法寫成可重複套用的腳本（一次性的放 scratchpad；要長期保留就**自行建立** `.claude/harness/scripts/` 目錄——它預設不存在，找不到不是環境異常）或記進 `05_maintenance.md` 踩坑區，之後由便宜模型照抄套用 |

**同一件事最多兩輪**（一輪＝一次完整嘗試＋一次修正）。第三輪不存在，直接熔斷。

**錯誤軌跡（Error Log）格式**——升級時必附，讓救援模型不用重走冤枉路：
```
目標：（一句話）
已嘗試（依序）：1. …  2. …
錯誤原文（關鍵行，非全文）：…
已排除的假設：…
卡點猜測：…
```

## 隔離驗證（實作者不得自我驗收）

- 適用：影響線上行為的改動、C 層大改動、以及任何「使用者無法自行判斷對錯」的產出。
- 作法：完成後派「fresh-context 驗收員」（`general-purpose`）。**只給它 AC＋檔案路徑，不給實作過程的敘事**——知道「實作者怎麼想」的驗收員會被帶風向。
- 驗收員義務：親自 read-back 檔案、實跑可跑的檢查（`node --check`、WebFetch 線上頁、測資）、逐條核對 AC，結論每一條都要附證據（file:line 或指令輸出）。
- 驗收不過 → 退回實作（計入重試輪數，最多兩輪）→ 仍不過 → 熔斷問使用者。
- 品味類產出（文案、UI、命名）不搞多代理辯論自決：直接產 2–3 個選項交使用者選（`03_rubrics.md` R4）。多樣本評審只用於「有客觀對錯」的技術選型。
