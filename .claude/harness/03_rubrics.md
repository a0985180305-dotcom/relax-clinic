# 03｜判斷力外化矩陣（弱模型檢核表）

> 用途：把高階模型的直覺量化成可肉眼比對的清單。拿不準時，逐條對照，**不要靠感覺**。
> 所有範例都以本專案真實情境撰寫。

## R1｜「方向錯了，立刻停」的訊號（換路徑，不是原地重試）

出現**任何一條**就停手，回到問題原點重新診斷（「幾次算出局」的統一定義在 `02_orchestration.md`「名詞定義」）：

- [ ] 同一條錯誤訊息出現第 2 次（換寫法不算換路徑，錯誤沒變就是同一堵牆）
- [ ] 為了修 A，需要動到 3 個以上「原本好好的」檔案或區段
- [ ] 腦中出現「先把驗證／規則繞過去，之後再補」的念頭
- [ ] 需要修改「已上線驗證過」的代碼，新代碼才能動 → 高機率是你理解錯，不是舊代碼錯
- [ ] 對外部 API 的回傳形狀（LINE、Firestore、Cloudflare）開始用「猜」的
- [ ] 說不出「這次嘗試和上次差在哪、為什麼這次會不一樣」

**【完美正例】** 客戶端班表時段沒顯示 → 停下來先查三件事實：① WebFetch 線上 booking.html 確認部署版本是不是舊的；② 查 Firestore 是否真的有該調理師的班表資料；③ Grep `loadSchedule` 讀該區段 ±80 行。發現根本是「後台還沒設班表」→ 回報使用者去後台設定，一行代碼都沒改。

**【典型反例】** 班表時段沒顯示 → 直接重寫 booking.html 的渲染函式 → 沒效 → 再改 Worker 的 slots 邏輯 → 把已測通的覆寫模型改壞 → 想 push「修復」（被 guard hook 攔下）。結果：兩處好代碼被改壞，真正原因（沒設資料）從頭到尾沒查。

## R2｜「真正完成」的量化條件（全部打勾才准說完成）

- [ ] hooks 全綠：檔案落地＋JS 語法通過（hook 報錯＝直接不合格）
- [ ] 新／改的行為**實跑過或線上查證過**（node 跑測資、WebFetch/curl 打端點），不是「看代碼覺得對」
- [ ] `git commit` 已建立（有還原點）
- [ ] 影響線上行為 → 已附「使用者 3 步驗證清單」，每步都是使用者用手機或瀏覽器就能做的動作
- [ ] 回報中明確列出「還沒做／做不到／未查證」清單（可以是空的，但必須寫）

**【完美正例】** 「已在 index.html 加上匯出按鈕（`index.html:1420` 附近），`node --check` 通過、已 commit `abc1234`。**尚未上線**——push 指令已備好等你確認。你驗證：① 等我 push 後打開後台 ② 客戶列表右上角應出現『匯出』③ 點了會下載一個 .json 檔。未做：舊瀏覽器相容性未測。」

**【典型反例】** 「已加上匯出功能，理論上點擊就會下載，應該可以了。」——沒證據、沒 commit、沒驗證步驟、還用了「理論上／應該」。這句話在本專案等於「未完成」。

## R3｜熔斷提問時機（停止自主作業，問使用者）

出現任何一條，立刻停下並用白話向使用者提問，**不要自己選一個繼續做**：

- [ ] 要動線上系統（push／deploy／rules／LINE 設定）
- [ ] 要刪除資料，或動到「客戶點數、效期、金額」相關邏輯且規格有疑義
- [ ] 規格有兩種合理解讀，且選錯的重工成本超過 5 分鐘
- [ ] 屬於品味／文案／定價（→ 走 R4 程序）
- [ ] 同一件事已重試兩輪（`02_orchestration.md` 的硬上限）
- [ ] 發現使用者的請求會違反紅線（例如要求把病歷資料放上公開頁面）——說明原因並給替代方案

**【完美正例】** 實作效期邏輯時發現邊界疑義：買10送2 效期 1 個月，「1/31 購買」到期日是 2/28 還是 3/2？→ 停手，列出兩種解讀與各自後果（客戶多 2 天 vs 少 2 天），請使用者拍板，拿到答案才寫碼。

**【典型反例】** 自己選了「加 30 天」寫進代碼，回報時隻字未提。三個月後客戶投訴效期算錯，沒人知道規則是 AI 私自決定的。

## R4｜品味極限條款（弱模型的天花板，硬性程序）

以下領域**禁止自行拍板**：UI 配色與佈局美感、對客戶的文案語氣、價格與方案設計、品牌措辭、圖文選單視覺。

程序（固定三步）：
1. 產出 **2–3 個完整可比較的選項**（能給預覽或截圖就給，文案就給全文不給大綱）
2. 每個選項附**一句話 trade-off**（白話，不用術語）
3. 使用者選定後照做，不再「順手優化」

**【完美正例】** 「預約成功通知的措辭，給你三版：A『已為您保留 7/10 14:00 鈞老師時段』（正式）；B『預約好囉！7/10 下午 2 點見～』（親切）；C『✅ 7/10 14:00｜鈞｜到店前 10 分鐘提醒您』（資訊型）。A 穩重但距離感、B 親近但少資訊、C 清楚但冷。你選哪個？」

**【典型反例】** 自認 B 最好直接寫進推播上線。語氣是店家品牌的一部分，AI 沒有資格替店主決定她跟客戶說話的方式。
