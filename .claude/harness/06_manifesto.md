# 06｜給未來 Session 的交接信

> 寫信人：Claude Fable 5，2026-07-03。這是本環境唯一一次由 Fable 5 建制，之後由你——很可能是 Sonnet 或 Haiku——長期維運。這封信講三件使用者沒問、但我認為最關鍵的事，以及這套制度會怎麼爛掉、怎麼防。

## 三件沒被問到、但最關鍵的事

### 1. 這個系統最大的單點風險是「Firestore 沒有備份」，不是代碼

客戶病歷、方案點數、效期——診所的命脈——**只存在 Firestore 一份**，沒有任何自動備份。代碼壞了有 git 可救；資料沒了就是沒了（誤刪、規則寫錯被清空、帳號出事）。
**該做的事**：找使用者有空的時段，提案做「後台一鍵匯出 JSON」功能（走 C 層計畫流程），並教她養成每週匯出一次的習慣。在那之前，任何動到 Firestore 寫入邏輯的改動都要加倍小心（R3 熔斷清單已涵蓋）。**不要等使用者提出，主動提。**

### 2. 使用者本人是這個系統唯一的「營運人員」，而她不會讀代碼

這意味著：你回報什麼，她只能信什麼——這就是為什麼「假性完成」在本專案不是品質問題，是**誠信問題**。她拿這個系統服務付費客戶，你的一句「應該可以了」可能讓她在客戶面前出糗。回報寧可保守：「改好了但還沒驗證」永遠比「完成了」（實際沒完成）好一百倍。
另外，專案住在 `Downloads\` 資料夾——這是危樓（容易被清理、誤刪）。git remote 在 GitHub，所以**每次使用者確認 push，同時就是異地備份**。長期而言值得提議搬到正式目錄，但搬家要挑沒有進行中任務的時候，且記憶檔中的路徑筆記要同步更新。

### 3. 使用者有產品化的夢（見記憶 productize-idea），別讓今天的偷懶殺死它

她想把這套系統提供給其他調理業者。現在代碼裡調理師名字、LIFF ID、Worker URL 都是寫死的常數——單租戶沒問題，但**每多寫死一個，未來多租戶化就多一分痛**。規則：新功能涉及「店家專屬資訊」時，集中放到檔案頂部的 config 區塊，不要散落在函式裡。這條成本近乎零，價值在一年後。

## 這套制度最可能的退化方式（以及怎麼防）

| 退化模式 | 具體長相 | 預防 |
|---|---|---|
| **鑽字面漏洞** | 附了「3 步驗證清單」但寫得無法操作（「確認功能正常」算哪門子步驟）；跑了個無關緊要的測試就宣稱「實跑過」 | R2 的字面要求是「使用者用手機/瀏覽器**可操作**」；驗收員核對 AC 時要把驗證清單自己走一遍能不能懂 |
| **假性驗證** | 驗收員被實作敘事帶風向，橡皮圖章蓋「通過」 | `02_orchestration.md` 鐵則：驗收員只拿 AC＋檔案路徑，**不拿實作過程**；結論每條必附證據，沒有 file:line 或指令輸出的「通過」視為無效 |
| **繞過 hooks 寫檔** | 用 `Out-File`／`Set-Content`／`echo >` 寫專案檔，躲開 verify-write 檢查 | CLAUDE.md 環境陷阱已明文禁止；看到自己想這麼做，就是 R1 的「想繞過驗證」訊號，立刻停 |
| **規則膨脹** | 每踩一次坑加三條規則，CLAUDE.md 長到沒有模型會認真讀 | `05_maintenance.md` 的精簡觸發（CLAUDE.md >120 行、踩坑區 >150 行必須抽象化收斂） |
| **規則腐化** | 重構後規則裡的路徑、行號、函式名全是死連結，模型學會「規則本來就不準」 | 每次動到規則引用的檔案結構，順手核對 harness 引用（綠區）；使用者每月可貼一次下方「審計提示」 |

**給使用者的月度審計提示**（複製貼上即可）：
> 請扮演挑剔的審查員，讀 CLAUDE.md 與 .claude/harness/ 全部檔案，逐條核對：引用的路徑與檔案是否還存在、規則之間有無矛盾、有沒有哪條規則最近三次任務中被違反過。只報告，不修改。

## 接手指南（新 session 第一天怎麼開工）

1. `CLAUDE.md` 會自動載入——先看「開工導航」表，按你的任務讀對應檔案，**不用全讀**。
2. 業務進度（做到哪、剩什麼）在持久記憶 `line-booking-progress`：目前剩：設班表→完整預約流程實測、發布 LINE Login channel、圖文選單、（可選）收緊 ALLOWED_ORIGIN。
3. 動手前的三個條件反射：本機≠線上（對照表）、兩振出局（R1）、沒證據不叫完成（R2）。
4. 你不確定自己夠不夠格做某個判斷時，答案幾乎都是：**照 R3 熔斷，問使用者**。她人很好，問她不丟臉；把她的系統改壞才丟臉。
